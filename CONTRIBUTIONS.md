# Contributing to Ram-G 🛠️

Thank you for wanting to contribute to **Ram-G: Grounded 21st-Century Epic Action-Adventure**! 

To maintain high code quality, biological and physical realism, and robust Object-Oriented design, please read and follow these contribution guidelines.

---

## 1. Technical Framework & OOP Paradigm

All gameplay systems are built upon **Unreal Engine 5.4 C++** and must adhere strictly to established Object-Oriented Programming (OOP) paradigms:

- **Encapsulation**: Limit access to internal variables (like biological rates, cardiac flow, and ATP reserves). Maintain protected/private scope and expose public configurations via `UPROPERTY` specifiers and controlled, read-only `UFUNCTION(BlueprintPure)` getters.
- **Inheritance Hierarchy**: Extends the core framework rather than rewriting it. Subclass specialized characters (e.g., `AHanumanCharacter`) from the foundational base character `ARamGCharacter`.
- **Polymorphism**: Override virtual functions cleanly (such as `BeginPlay()`, `Tick()`, and `SetupPlayerInputComponent()`) to inject custom mechanics while preserving base capabilities using `Super::FunctionName()`.

### Code Style Guide
- **Naming Conventions**: Unreal standard PascalCase for classes (`AHanumanCharacter`, `URamGAstraComponent`, `EMission0State`). Use camelCase for local variable contexts. Boolean variables must start with a lowercase `b` (e.g., `bIsExhausted`).
- **UPROPERTIES**: Always specify categories and descriptive tooltips:
  ```cpp
  UPROPERTY(EditAnywhere, BlueprintReadOnly, Category = "RamG | Physiology", meta = (ToolTip = "Sanskrit identifier for epic integration"))
  FString SanskritName;
  ```
- **Robust Documentation**: Inline comments are required for complex mathematical models, physics calculations (such as spring grappling constants), or timing delays.

## 2. Physiological & Biomechanical Consistency

Our design is grounded in realistic, scientific limitations of the body. When designing characters, abilities, or upgrades, you must align with the **Grounded Biomechanics Specifications**:

| Metric / Pool | Default Human Base | Divine/Avatar Base | Gameplay Application |
| :--- | :--- | :--- | :--- |
| **Stamina** | 100% | 100% (High capacity) | Depletes on dashing or swinging; regenerated by breathing exercises (*Pranayama*). |
| **Neurological Latency** | ~200ms | 0ms - 50ms | Input latency simulation, dictating reaction time offsets. |
| **Max Cardiac Output** | ~25 L/min | ~60 L/min | Limits stamina recovery rates and thresholds for high fatigue. |
| **Lactic Acid Threshold**| 4.0 mmol/L | 8.0 mmol/L | Fatigue marker. If exceeded, walk speed is restricted to a heavy crawl. |
| **ATP Recovery Rate** | 10 J/s | 50 J/s | Fast recovery limits for energetic maneuvers and heavy combat combos. |

## 3. Pull Request and Git Guidelines

We enforce a clean git lineage:
1. **Create a Feature Branch**: Keep branches focused and descriptive (e.g., `feature/airavat-shockwaves` or `fix/grapple-tension`).
2. **Submit a Detailed Pull Request**: Utilize our official [Pull Request Template](.github/pull_request_template.md).
3. **Link Your Issue**: Ensure your PR links to a relevant Issue (e.g., Bug Report or Feature Request).

---

## 4. Run Playtests & Diagnostics

Before proposing any changes, verify code reliability and mechanical balance:
1. Make sure your C++ code compiles successfully in your IDE (VS2022 / Rider).
2. Launch the Unreal Engine 5.4 Editor.
3. Call `AMission0GameMode::RunMissionDiagnostic()` via console command or blueprints.
4. Verify that all 5 baseline checks (Hanuman Active, Double Jump configured, Grapple limit check, Fruit Basket storage incrementing, and Vajra Nerf biological locks) report `[PASS]`.
5. Attach the diagnostic output logs to your PR description.
